# AI 教学媒体平台产品需求文档

> 文档状态：**已确认**（角色体系 / 注册 / 学科审核范围）  
> 确认日期：2026-07-21  
> 关联：`docs/product.md`、`docs/data-model.md`、`docs/api.md`  
> 说明：§14 产品决策已锁定；**实现阶段再改代码**。

## 1. 定位与目标

平台把 `ai-teaching-media` 的六个子能力包装成一套面向 K12 **学生、教师、管理员** 的生产与分享系统：用户输入知识点或文章，系统异步生成信息图、教学视频、章节解说视频、技术文章插图和短视频封面，并通过课程分类、**角色化审核** 与公开广场分发。

### 1.1 本轮需求升级目标

当前实现过于扁平（基本只有 `user` / `admin`）。本轮需求明确为：

1. 引入 **学生 / 教师 / 管理员** 三类正式角色。
2. 教师可 **审核学生作品**；审核范围受 **授课学科** 约束。
3. 管理员拥有 **最高权限**，可跨学科查看与管理。
4. 系统提供 **用户注册**：注册时选择角色；教师必须选择授课学科。

## 2. 用户角色与权限（RBAC）

### 2.1 角色定义

| 角色 code | 中文名 | 说明 |
|---|---|---|
| `student` | 学生 | 学习用户；可浏览公开课程、提交自己的生成任务与作品送审 |
| `teacher` | 教师 | 授课用户；可生成教学媒体；可审核 **本学科范围内** 的学生作品 |
| `admin` | 管理员 | 平台运营/超管；跨学科、跨用户的最高权限 |

> 兼容说明：历史数据中的 `user` **一律迁移为 `student`**（见 §10 / §14）。新注册仅 `student` / `teacher`。

### 2.2 能力矩阵

| 能力 | 学生 | 教师 | 管理员 |
|---|:---:|:---:|:---:|
| 注册 / 登录 / 退出 | ✅ | ✅ | ✅（通常预置，不开放公开自助注册） |
| 浏览课程广场（已审核公开课） | ✅ | ✅ | ✅ |
| 创建生成任务（**全部 7 档位**） | ✅ | ✅ | ✅ |
| 查看自己的任务 / 资产 / 草稿课 | ✅ | ✅ | ✅ |
| 编辑自己的课程草稿并送审 | ✅ | ✅ | ✅ |
| 审核 **学生** 提交的作品 | ❌ | ✅（限本学科） | ✅（全部） |
| 审核 **教师** 提交的作品 | ❌ | ❌ | ✅（**仅管理员**） |
| 查看待审列表（跨作者） | ❌ | ✅（学科过滤） | ✅（全量） |
| 下架 / 驳回任意课程 | ❌ | 限本学科且原作者为学生 | ✅ |
| 用户管理（启用/禁用、改角色） | ❌ | ❌ | ✅ |
| 系统配置（TTS/生图 provider 等） | ❌ | ❌ | ✅ |
| 全站任务运维（重试/取消任意任务） | ❌ | ❌ | ✅ |
| 查看统计看板 | ❌ | 可选：本学科汇总 | ✅ 全站 |
| 自改授课学科（≥1） | ❌ | ✅ | 可代改任意用户 |

### 2.3 学科范围规则（核心）

1. **教师授课学科**：注册/资料中维护 `teacher_subjects[]`（如 `physics`、`math`）。至少 1 个学科；**教师可自行修改**，保存后仍须 ≥1。
2. **作品学科**：取自课程/任务的 `subject`（与现有 catalog 学科 code 一致）。
3. **教师可见待审作品** 必须同时满足：
   - 作品 `publish_status = pending`（或审核队列可见状态）
   - 作品作者角色为 `student`
   - 作品 `subject ∈ 教师.teacher_subjects`
4. **非对应学科教师不可见**：例如物理学生作品，数学教师在待审列表中看不到该行；打开详情或审核 API **统一返回 HTTP 404**（防枚举/信息泄露）。
5. **管理员不受学科限制**，可查看全部学生与教师作品。
6. 教师 **不能** 审核其他教师作品；**教师作品仅由管理员审核**。
7. 学生 **不能** 审核任何作品。
8. **无对口教师兜底**：若某学科当前不存在 `status=active` 且 `teacher_subjects` 含该学科的教师，则该科学生待审 **进入管理员队列**（管理员可见可审）。

### 2.4 作者与所有权

| 对象 | 学生 | 教师 | 管理员 |
|---|---|---|---|
| 自己的 job/course/asset | 读写（自身范围） | 读写（自身范围） | 读写任意 |
| 他人私有草稿 | 不可见 | 不可见（即使同学段） | 可见 |
| 公开已通过课程 | 可见 | 可见 | 可见 |

### 2.5 审核策略（首期固定值，已确认）

| 配置键 | 首期固定值 | 说明 |
|---|---|---|
| `review.student_courses_by` | `subject_teacher` | 学生作品由对应学科教师审核 |
| `review.teacher_courses_by` | `admin` | **教师作品仅管理员审核** |
| `review.fallback_to_admin` | `true` | **无对口教师时，学生待审进入管理员队列** |
| `review.allow_teacher_self_publish` | `false` | 教师作品不可免审直发 |
| `review.teacher_invite_code_required` | `false` | **首期教师注册不要邀请码** |

> 上表为首期产品决策锁定值；实现可先写死，后续再做成管理员可配置项需另开需求。

## 3. 注册与登录

### 3.1 注册入口

- Web：**注册**页（与登录并列）
- API：扩展现有 `POST /api/auth/register`

### 3.2 注册字段

| 字段 | 学生 | 教师 | 管理员 | 校验 |
|---|---|---|---|---|
| email | 必填 | 必填 | 不开放公开注册 | 邮箱格式、全局唯一 |
| password | 必填 | 必填 | — | ≥ 8 位，含字母+数字（可配置） |
| nickname | 必填 | 必填 | — | 2–32 字 |
| role | 必填，`student`/`teacher` | 同左 | — | 仅允许学生/教师；**禁止**注册为 `admin` |
| teacher_subjects | 不填 | **必填，≥1** | — | 必须属于系统学科字典 |
| grade（可选，学生） | 可选 | 不填 | — | 如 `grade8`，便于推荐 |
| invite_code | 不使用 | 不使用 | — | **首期不做邀请码**；不要求、不校验 |

### 3.3 注册业务规则

1. 公开注册 **仅** `student`、`teacher`。
2. `admin` 仅能由：
   - 环境变量初始化（现有 `ADMIN_EMAIL`），或
   - 已登录管理员在后台创建/提权。
3. 选择 `teacher` 但未选学科 → 注册失败，提示「请至少选择一个授课学科」。
4. 选择 `student` 却提交学科字段 → 首期忽略，不入库为审核学科。
5. 注册成功后默认 **自动登录**（返回 sessionToken）。
6. 注册即 `status=active`；首期不做「教师人工开通」。
7. **不校验邀请码**（已确认）。

### 3.4 登录

- 保持邮箱 + 密码。
- 登录响应与 `GET /auth/me` 必须返回：`id/email/nickname/role/teacherSubjects/status`。
- 前端按 role 展示菜单：
  - 学生：生成、我的任务、我的作品、广场
  - 教师：以上 + **待我审核** + 审核记录 + 个人中心授课学科
  - 管理员：以上 + 用户管理 + 全站审核 + 系统配置 + 任务运维

### 3.5 资料维护（注册后）

| 操作 | 学生 | 教师 | 管理员 |
|---|---|---|---|
| 改昵称/密码 | ✅ | ✅ | ✅ |
| 改授课学科 | ❌ | ✅ **允许自改**，保存后仍须 ≥1 个学科 | 可代改任意用户 |
| 自助改 role | ❌ | ❌ | 仅管理员改他人 |

## 4. 功能范围

### 4.1 媒体生产（沿用）

| 输出档位 | 输入 | 主要产物 |
|---|---|---|
| `teaching_video_full` | 学科、年级、知识点、学习目标 | storyboard、分段音频、1080p MP4、元数据 |
| `infographic_only` | 知识点与学科上下文 | 9:16 PNG、prompt、生成元数据 |
| `package_all` | 知识点与课程信息 | 教学视频 + 9:16 信息图 + 3:4 封面 + storyboard |
| `tech_article_diagram` | Markdown 文章、风格 | 4-5 张 16:9 插图及位置清单 |
| `article_explainer_video` | Markdown 文章、主题、语音 | 章节 storyboard、配音、1080p MP4 |
| `short_video_cover` | 标题、口播文案、参考图（可选） | 3:4 PNG、prompt、元数据 |
| `image_generation` | 提示词/参考图 | 通用 PNG |

> **学生与教师均可创建全部生成档位**（已确认）；权限差异在 **审核与运营**，不在生成能力。

### 4.2 课程与作品流

1. 生成成功 → 可选自动创建 **私有课程草稿**（`visibility=private`, `publish_status=draft`）。
2. 作者完善标题/摘要/标签后 **送审** → `pending`。
3. 审核人：
   - 学生作品 → 对应学科教师；**若该学科无在职教师 → 进入管理员队列**
   - 教师作品 → **仅管理员**
4. 审核通过 → `approved` + 可设 `visibility=public` 进入广场。
5. 审核驳回 → `rejected`，作者可见意见，可修改后再次送审。
6. 管理员可随时下架：`approved → rejected` 或 `visibility=private`。

### 4.3 教师审核工作台（新增）

- 待审列表：仅展示本学科学生作品
- 筛选：年级、时间、关键词
- 详情：视频/图片预览、作者昵称、学科年级、提交时间
- 操作：通过 / 驳回（意见选填或必填由实现定，建议驳回必填）
- 我的已审记录：本人历史审核动作
- 越权访问非本学科/非学生作品详情 → **404**

### 4.4 管理员工作台（增强）

- 用户管理：列表、禁用/启用、调整角色、维护教师学科
- 全站待审 / 已审课程（含教师作品、无对口教师的学生作品）
- 任务运维：任意任务重试/取消
- 系统配置：provider、（后续）审核策略
- 统计：全站任务量、各学科待审积压

### 4.5 课程广场

- 仅 `publish_status=approved` 且 `visibility=public`
- 支持学科、年级、关键词筛选
- 所有登录角色可浏览公开课；下载私有资产仍需鉴权

## 5. 业务规则

### 5.1 生成任务

1. `topic`、`subject`、`grade` 为知识点型任务必填；文章型任务必须提供 `article`。
2. 任务状态机：`queued → running → succeeded|failed|cancelled`；失败可重试。
3. 依赖缺失不得伪造成功；错误写入 `error_message`。
4. API Key 仅服务端环境变量；前端/DB 不存明文 Key。
5. 用户只能操作自己的任务；管理员可操作全部。
6. **学生可使用全部 7 个 outputProfile**，服务端不对角色做档位收紧。

### 5.2 审核与可见性

1. 公开课程必须同时满足 `visibility=public` 且 `publish_status=approved`。
2. 教师审核 API 必须在服务端按学科与作者角色二次校验。
3. 教师越权（学科不匹配 / 作者非学生 / 审教师作品）→ **HTTP 404**（不是 403）。
4. 审核写 `course_reviews`：审核人、角色、动作、意见、时间、课程学科快照。
5. 同一课程在 `pending` 期间可被有权限审核人处理；管理员可覆盖处理。
6. 驳回意见对作者可见；对其他学生不可见。
7. 无对口教师时，学生 pending 课必须出现在管理员待审列表。

### 5.3 安全与合规

1. 禁止水平越权：教师不得通过猜 ID 读取非本学科学生私有/待审详情。
2. 禁用用户（`status=disabled`）不可登录；已公开内容默认保留。
3. 密码哈希沿用 scrypt；会话 Bearer token + 过期。

## 7. 接口增量（需求层）

### 7.1 Auth

`POST /api/auth/register`

```json
{
  "email": "stu@example.com",
  "password": "Passw0rd",
  "nickname": "小明",
  "role": "student",
  "grade": "grade8"
}
```

教师示例：

```json
{
  "email": "physics.teacher@example.com",
  "password": "Passw0rd",
  "nickname": "王老师",
  "role": "teacher",
  "teacherSubjects": ["physics"]
}
```

规则：

- `role=admin` → 400
- `role=teacher` 且 subjects 为空 → 400
- 不要求 `inviteCode`
- 成功：201 + user 摘要 + token（自动登录）

`GET /api/auth/me` 增加 `teacherSubjects`。

教师自改学科（建议）：

- `PATCH /api/me/profile` `{ "nickname"?: "...", "teacherSubjects"?: ["physics","math"] }`  
  教师提交空学科数组 → 400

### 7.2 教师审核

- `GET /api/teacher/reviews/pending?subject=&grade=&page=`  
  仅本学科学生 pending 课
- `GET /api/teacher/reviews/done?page=`  
  本人已审
- `POST /api/teacher/courses/:id/review`  
  body: `{ "action":"approve|reject", "comment":"..." }`  
  服务端校验：作者为 student 且 subject ∈ teacherSubjects；否则 **404**

### 7.3 管理员

- 保留现有 `/api/admin/*`
- 新增/增强：
  - `GET /api/admin/users`
  - `PATCH /api/admin/users/:id`（role/status/teacherSubjects）
  - 审核接口可审全部角色作品
  - 待审列表包含：教师作品 + 无对口教师的学生作品 + 有对口教师的学生作品（全量）

### 7.4 错误码约定

| HTTP | 场景 |
|---|---|
| 400 | 注册字段非法、教师无学科、教师自改后学科为空 |
| 401 | 未登录 |
| 403 | 角色不足访问管理端等明确越权场景（非待审资源隐藏） |
| 404 | 资源不存在，**或** 教师无权查看/审核的课程详情（**统一 404**） |

## 9. 用户故事（验收用）

1. **学生注册生成**  
   小明注册为学生（初二），提交「物理 / 能量守恒定律」视频任务（任意档位均可），成功后送审。
2. **对口教师审核**  
   物理老师登录，在待审列表看到小明的课；数学老师登录 **看不到** 该课。
3. **教师通过**  
   物理老师通过后，课程进入广场；小明作品状态为已通过。
4. **教师驳回**  
   驳回时填写「概念图公式有误」；小明可见意见并修改后再次送审。
5. **跨学科隔离**  
   物理老师打开数学学生 pending 课详情 → **404**。
6. **教师作品审级**  
   物理老师自己的作品送审后，**只出现在管理员队列**，不出现在其他教师待审列表。
7. **管理员兜底**  
   若某学科暂无在职教师，该科学生待审 **自动进入管理员队列**，管理员可审。
8. **管理员用户治理**  
   管理员可禁用违规账号、把用户角色从学生调整为教师并设置学科。
9. **禁止注册管理员 / 无邀请码**  
   注册 `role=admin` 失败；教师注册无需邀请码。
10. **教师自改学科**  
    教师把授课学科从仅物理改为物理+数学后，数学学生 pending 出现在其待审列表；改回仅物理后不再可见数学课。

## 10. 迁移与兼容

| 现状 | 目标（已确认） |
|---|---|
| `role=user` | **一律迁移为 `student`** |
| `role=admin` | 保持 `admin`，`teacher_subjects=[]` |
| 演示账号 `teacher@demo.local` | 保持 admin；建议另建示例教师 `physics.teacher@demo.local`（学科 physics） |
| 旧审核仅管理员 | 增加教师按学科审学生通道；管理员全量通道保留；**教师作品仍仅管理员审** |

迁移要求：

1. 不丢失历史课程与任务归属。
2. 上线脚本/启动迁移幂等：`role='user'` → `role='student'`（MySQL 与内存 JSON 均处理）。
3. 文档与 API 响应字段同步更新。

## 13. 验收标准（DoD）

实现后应满足：

1. 注册页可选学生/教师；教师必须选学科；**无邀请码**。
2. 三角色菜单与 API 权限一致。
3. 学生物理作品仅物理教师 + 管理员可见待审。
4. 教师不能审教师作品；管理员可以。
5. 无对口教师时，学生待审出现在管理员队列。
6. 审核记录可追溯。
7. 旧 `user` 数据迁移为 `student` 且可登录。
8. 教师可自改授课学科且 ≥1。
9. 越权待审详情统一 **404**。
10. 学生可创建全部生成档位。
11. 自动化测试覆盖：注册校验、学科隔离、迁移、越权 404。
